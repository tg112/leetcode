# Minimum Window Substring

## Point

文字列 s の中から、ターゲット t に含まれるすべての文字（重複も含む）をギリギリ包み込める、一番短い区間（窓）を切り取るという問題


### 1. 可変サイズ型スライディングウィンドウ（収縮型）
この問題は「条件を満たすまで右端（right）を広げ、満たした瞬間に左端（left）を限界まで削る」という動的な窓のコントロールが必要。
- **拡大フェーズ**: `formed < required` の間は、貪欲に `right` を右に動かす。
- **縮小（最適化）フェーズ**: `formed == required` になったら、条件を崩さないギリギリのラインまで `left` を右に動かして窓を狭め、最小の長さを記録する。

### 2. Frequency Map（出現回数ハッシュマップ）の正確な管理
`collections.Counter` と `defaultdict` を用いて、「目標の文字数」と「現在の窓の中の文字数」をリアルタイムに比較します。
- `window_counts[char] == dict_t[char]` になった瞬間だけ `formed` を `+1` する。
- 左端を削って `window_counts[char] < dict_t[char]` になった瞬間だけ `formed` を `-1` する。
※ `<=` や `>` ではなく、`==` や `<` の「境界をまたいだ瞬間」だけを捉えるのがバグらせないコツです。

### 3. スライス処理の最適化（最後にまとめて切り出す）
ループの途中で見つけた有効な文字列を毎回 `s[left:right+1]` のように実際に切り取って変数に保存すると、文字列のコピーに毎回 O(N) の時間がかかってタイムアウト（TLE）します。
提示されたコードのように、`min_left` と `min_right` という「インデックスの数字（整数）」だけを記憶しておき、**最後の最後に1度だけ切り出す**のが鉄則です。

## 計算量

- 時間計算量: 
    - O(|S| + |T|)最初に t のハッシュマップを作るのに O(|T|) かかります。
    - その後、right ポインタが s の末尾まで進む間に、left ポインタも最大で s の末尾までしか進みません。2つのポインタが配列をそれぞれ最大1周するだけなので、s の走査にかかる時間は O(|S|) です。全体の合計は O(|S| + |T|) となり、線形時間で動作する最高速の処理です。
- 空間計算量: O(|S| + |T|)
    - ハッシュマップ dict_t に最大で t のユニークな文字数分、window_counts に最大で s のユニークな文字数分のメモリを使用します（アルファベットの総数26や52種類に限定される場合は実質 O(1) とみなすことも可能です）。
